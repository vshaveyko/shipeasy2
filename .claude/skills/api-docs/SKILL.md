# Admin API documentation pipeline

Single source of truth: Zod schemas in `packages/core/src/schemas/`. Everything downstream — OpenAPI spec, individual MDX pages, the consolidated `/flags-experiments/api/` index — is generated. Edit the schema + resource descriptor, regenerate, ship.

The published consumer is the **docs site** (`shipeasy-docs` Worker, `apps/docs/`) and the **typed admin client** (`packages/admin-api/` consumed by `packages/cli` and `packages/mcp-server`). Both read the same `openapi.json`.

---

## Layered architecture

| Layer                  | File                                                               | What it owns                                                                                                                                                                              |
| ---------------------- | ------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1. Schemas             | `packages/core/src/schemas/<resource>.ts`                          | Zod request/response shapes. `.describe()` strings flow into rendered Description column. `.default()` shows as default chip.                                                             |
| 2. Resource descriptor | `packages/admin-api/src/resources/<resource>.ts`                   | Tag, basePath, per-endpoint `summary` / `description` / `useCase` markdown / `requestExamples` / `pathParams`. Imports schemas from layer 1.                                              |
| 3. Spec emit           | `packages/admin-api/scripts/emit-spec.ts`                          | Registers resources, writes `packages/admin-api/openapi.json`. Run `pnpm --filter @shipeasy/admin-api emit-openapi`.                                                                      |
| 4. Per-op MDX          | `apps/docs/content/docs/flags-experiments/api/operations/<op>.mdx` | One file per operation. Uses `<APIPage>` reading `openapi.json`. Generated by `apps/docs/scripts/generate-api-reference.ts`.                                                              |
| 5. Index page          | `apps/docs/content/docs/flags-experiments/api/index.mdx`           | Hosts `<ApiList />` — reads `openapi.json` directly at build time, renders the rich Body / Response / Use-cases tables with try-it. Component at `apps/docs/src/components/api-list.tsx`. |

## Add a new resource end-to-end

Example below pretends we're adding `configs`. Substitute your resource name.

### 1. Author the schema (`packages/core/src/schemas/configs.ts`)

```ts
import { z } from "zod";

export const configCreateSchema = z
  .object({
    name: z
      .string()
      .regex(/^[a-z0-9][a-z0-9_-]{0,63}$/)
      .describe("Stable config key used by SDKs. Lowercase letters, digits, `_` or `-`."),
    value: z
      .unknown()
      .describe("Default JSON value returned to callers that don't match any rule."),
    rules: z
      .array(/* … */)
      .default([])
      .describe("Targeting predicates evaluated in order; first match wins."),
  })
  .describe("Body for `POST /api/admin/configs`. At minimum supply `name` + `value`.");

export const configUpdateSchema = z
  .object({
    /* … */
  })
  .describe("Body for PATCH.");

// Response shapes — used by the spec generator. Runtime handlers don't .parse() these.
export const configResponseSchema = z.object({
  id: z.string().describe("Opaque config id."),
  name: z.string(),
  value: z.unknown(),
  updatedAt: z.string().describe("ISO-8601 timestamp."),
});
export const configCreateResponseSchema = z.object({ id: z.string(), name: z.string() });
export const configUpdateResponseSchema = z.object({ id: z.string() });
export const configDeleteResponseSchema = z.object({ ok: z.literal(true) });
```

Rules:

- **`.describe()` on every field** — this is what populates the Description column in `<ApiList />` and the Fumadocs APIPage. No describe = empty `—` cell.
- **`.default(...)` for optionals with a fallback** — `<ApiList />` extracts it and shows `default: <value>` next to the description. Fields with a `default` are NOT marked `required` in the rendered table even if they appear in JSON Schema's `required[]`.
- **Top-level `.describe()` on the request schema** — sets the schema's `description`, shown by APIPage above the body fields.
- **Naming**: `<resource>CreateSchema`, `<resource>UpdateSchema`, `<resource>ResponseSchema`, `<resource>{Create,Update,Delete,Toggle}ResponseSchema`. Resource descriptor imports by these exact names.

### 2. Resource descriptor (`packages/admin-api/src/resources/configs.ts`)

Mirror `packages/admin-api/src/resources/gates.ts`. The shape:

```ts
export const configsResource = {
  name: "configs" as const,
  basePath: "/api/admin/configs",
  describeOne: "dynamic config",
  describeMany: "dynamic configs",
  tag: {
    name: "Configs",
    description: [
      "One-paragraph overview shown on the API landing page above the operation list.",
      "",
      "**Use markdown.** Bold, code, lists all render in the docs sidebar tag header.",
    ].join("\n"),
  },
  schemas: { create: configCreateSchema, update: configUpdateSchema },
  actions: [] as const, // Custom POST <basePath>/:id/<action> verbs. Use [] if none.
  endpoints: [
    {
      operationId: "listConfigs",
      method: "GET",
      path: "",
      summary: "List dynamic configs",
      description: "Returns a single page ordered by `updated_at desc, id desc`.",
      response: pageOf(configResponseSchema), // pageEnvelopeSchema helper, see gates.ts
      examples: {
        response: {
          data: [
            /* … */
          ],
          next_cursor: null,
        },
      },
      useCase: "Snapshot every config in the project — e.g. CI check for stale defaults.",
    },
    {
      operationId: "createConfig",
      method: "POST",
      path: "",
      summary: "Create a dynamic config",
      description: [
        "Top-line description rendered above the Body table.",
        "",
        "**Use cases**", // ← this marker triggers the rich Use cases table.
        "",
        "- **Dark create** — supply only `name` + `value`. Ramp via PATCH later.",
        "- **Targeted defaults** — attach `rules` to vary the value per user attribute.",
      ].join("\n"),
      successStatus: 201,
      request: configCreateSchema,
      response: configCreateResponseSchema,
      examples: {
        // KEYS matter — the use-case-to-example matcher tokenizes the key + summary
        // and intersects with bullet title tokens. Pick keys/summaries that share
        // words with the bullet titles.
        requestExamples: {
          dark: {
            summary: "Minimal — dark config, name + value only",
            description: "Smallest valid body. Ramp by attaching rules later.",
            value: { name: "checkout_copy", value: { headline: "Checkout" } },
          },
          targeted: {
            summary: "Targeted — varied defaults by plan",
            description: "Attach a rules array so Pro users get a different value.",
            value: {
              /* … */
            },
          },
        },
        response: { id: "cfg_…", name: "checkout_copy" },
      },
      useCase: [
        '- **Dark create** — `{ "name": "…", "value": … }`. Ramp via PATCH.',
        "- **Targeted defaults** — supply `rules` to vary the value per user.",
      ].join("\n"),
    },
    // updateConfig (PATCH /{id}), deleteConfig (DELETE /{id}), and any custom actions…
  ] as const,
} as const;
```

How fields drive the rendered output:

| Resource field                                       | Where it surfaces                                                                                                                                                                                                                                                                  |
| ---------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `tag.name` / `tag.description`                       | OpenAPI `tags[].description`; sidebar section header on the index page.                                                                                                                                                                                                            |
| `endpoints[].summary`                                | `<h3>` in the operation block + sidebar label.                                                                                                                                                                                                                                     |
| `endpoints[].description`                            | Top description block. **If it contains `**Use cases**`followed by a`- **Title** — body` bullet list, that block is parsed out and rendered as the 3-column Use cases table; the intro before the marker stays as plain description.**                                             |
| `endpoints[].useCase`                                | Legacy single-sentence "Use case:" callout (set this OR the `**Use cases**` markdown in description, not both).                                                                                                                                                                    |
| `endpoints[].request`                                | Zod schema → JSON Schema → Body table (Name / Type / Description columns). Nested object shapes inline up to depth 2; enums up to 12 values render inline (`"eq" \| "neq" \| …`).                                                                                                  |
| `endpoints[].response`                               | Same pipeline → Response table.                                                                                                                                                                                                                                                    |
| `endpoints[].pathParams`                             | `{ id: "description" }` → Path Parameters rows.                                                                                                                                                                                                                                    |
| `endpoints[].examples.requestExamples`               | Map → openapi `examples`. First entry becomes default code-sample / try-it body. Each entry **may** bind to a Use cases row via token overlap (key + summary tokens vs bullet title tokens, score ≥ 1 wins). Pick keys that share words with bullet titles or examples render "—". |
| `endpoints[].examples.response` / `responseExamples` | OpenAPI example for the success response. Shown as JSON below the Response table.                                                                                                                                                                                                  |
| `endpoints[].successStatus`                          | Defaults: POST → 201, others → 200.                                                                                                                                                                                                                                                |

### 3. Register the resource (`packages/admin-api/scripts/emit-spec.ts`)

Append to the `resources` array:

```ts
import { configsResource } from "../src/resources/configs.js";
// …
const doc = buildOpenApi({
  info: {
    /* … */
  },
  servers: [
    /* … */
  ],
  resources: [gatesResource, configsResource], // ← add here
});
```

### 4. Regenerate the spec

```bash
pnpm --filter @shipeasy/admin-api emit-openapi
```

This writes `packages/admin-api/openapi.json`. Commit the regenerated file alongside the schema/resource edits — the docs build (and `<ApiList />` runtime import) consume it directly.

### 5. Generate / hand-write per-operation MDX pages

The repo includes a generator: `apps/docs/scripts/generate-api-reference.ts` (writes one `.mdx` per `endpoints[]` entry under `apps/docs/content/docs/flags-experiments/api/operations/`). Run it (check `apps/docs/package.json` for the script name) or hand-create files matching the existing pattern:

```mdx
---
title: Create a config
full: true
---

Intro paragraph (mirrors the description in the resource descriptor).

<APIPage
  document={"/abs/path/to/packages/admin-api/openapi.json"}
  operations={[{ path: "/api/admin/configs", method: "post" }]}
/>
```

Then update `apps/docs/content/docs/flags-experiments/api/operations/meta.json`:

```json
{ "pages": ["listConfigs", "createConfig", "updateConfig", "deleteConfig"] }
```

### 6. Verify the rendered output

```bash
pnpm --filter @shipeasy/docs dev
```

Then:

- **Consolidated page**: http://localhost:3000/flags-experiments/api/ — should show your new tag section with operation cards, each with Body / Response / Use cases tables and Try it.
- **Per-op page**: http://localhost:3000/flags-experiments/api/operations/createConfig/ — Fumadocs APIPage view (Header Parameters, Request Body, Response Body sections).

If a body field is missing from the rendered table:

- Confirm it has a `.describe()` and is part of the Zod object (not stripped by `.omit()`/`.pick()`).
- Re-run `emit-openapi` and check `packages/admin-api/openapi.json` → `components.schemas.<Op>Request.properties`.

If a Use cases row's example shows "—":

- Add tokens to the example key or `summary` that overlap with the bullet title. Matcher = `apps/docs/src/components/api-list.tsx` → `matchExampleKey` (token overlap, ≥ 3 chars, score ≥ 1 wins).

### 7. Ship

```bash
git add packages/core/src/schemas/configs.ts \
        packages/admin-api/src/resources/configs.ts \
        packages/admin-api/scripts/emit-spec.ts \
        packages/admin-api/openapi.json \
        apps/docs/content/docs/flags-experiments/api/operations/
git commit -m "docs(api): add configs resource to admin API reference"
```

Cloudflare Workers Builds redeploys `shipeasy-docs` on push to `main`. No manual deploy.

## Add a single field to an existing resource

The 90% case. Don't re-derive the whole pipeline:

1. Add the field to the Zod schema in `packages/core/src/schemas/<resource>.ts` with `.describe()` (and `.default()` if optional-with-fallback).
2. If the API handler in `apps/ui/src/lib/handlers/<resource>.ts` needs to persist it, add the patch line there too (e.g. `if (parsed.new_field !== undefined) patch.newField = parsed.new_field;`).
3. `pnpm --filter @shipeasy/admin-api emit-openapi`.
4. Done. Per-op MDX pages re-render the new field automatically (they reference `openapi.json` at build time).

No MDX edit, no resource descriptor edit, no use-case touch — unless the field warrants its own use-case bullet.

## Reference snippets (copy-paste)

### Page envelope for list endpoints

```ts
import { z } from "zod";
import { configResponseSchema } from "@shipeasy/core/schemas/configs";

const configListResponseSchema = z.object({
  data: z.array(configResponseSchema),
  next_cursor: z.string().nullable(),
});
```

The OpenAPI builder auto-adds `?limit` / `?cursor` query params when the response schema looks like a page envelope (`data: array` + `next_cursor`).

### Custom action (POST /{id}/<verb>)

Inside the resource descriptor:

```ts
actions: [
  { name: "publish", description: "Publish a draft config" },
] as const,
endpoints: [
  // …
  {
    operationId: "publishConfig",
    method: "POST",
    path: "/{id}/publish",
    summary: "Publish a config",
    description: "Promotes the working copy to live. Idempotent.",
    pathParams: { id: "Opaque config id (`cfg_…`)." },
    response: z.object({ id: z.string(), published: z.literal(true) }),
    examples: { response: { id: "cfg_…", published: true } },
    useCase: "Cut a release of a draft config without touching its rules.",
  },
],
```

### Description with both intro and use-cases table

```ts
description: [
  "First paragraph rendered as plain prose above the tables.",
  "",
  "Second paragraph still in the intro.",
  "",
  "**Use cases**",
  "",
  "- **Title one** — description after the em-dash; backticks `like this` render as code; **bold** works.",
  "- **Title two** — second case.",
].join("\n"),
```

The parser at `parseUseCases` in `apps/docs/src/components/api-list.tsx` keys off the literal string `**Use cases**` and the bullet pattern `/^\s*-\s+\*\*(.+?)\*\*\s*[—-]\s*(.+)$/`. Stick to this shape exactly or rows won't extract.

## Hard rules

1. **Never edit `packages/admin-api/openapi.json` by hand.** Regenerate via `emit-openapi`. Hand edits get clobbered.
2. **Never skip `.describe()`.** Empty descriptions ship as `—` cells in the docs.
3. **Never publish docs from a local build.** Cloudflare Workers Builds owns `shipeasy-docs` deploys (see the `infra` skill / `DEPLOY.md`). Push to `main` and CF rebuilds.
4. **Mark fields with `.default()` as optional in intent.** The renderer treats `required[] ∩ !hasDefault` as "must send"; everything else displays as optional with a `default:` chip. Don't game the JSON Schema `required` array.
5. **Match use-case bullet titles to example keys.** If a row in the Use cases table shows "—", the matcher couldn't bind. Either rename the example key/summary to share a ≥ 3-char token with the bullet title, or accept the dash.
