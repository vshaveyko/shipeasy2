# D1 Schema

Source of truth: `packages/core/src/db/schema.ts` (Drizzle schema definitions, shared between Next.js and Worker).
SQL migrations are generated from it via `drizzle-kit generate` and applied with `wrangler d1 migrations apply`.
The SQL below documents the table structure; the TypeScript schema is what actually drives the database.

## Plans — YAML config, NOT a D1 table

Plans live in `packages/core/src/config/plans.yaml`, compiled into both the Next.js app and Worker bundles at build time. The `projects.plan` column is a plain string (no FK). To change limits: edit `plans.yaml`, redeploy both Next.js and Worker — no migration, no KV write.

See `02-kv-cache.md` and `03-worker-endpoints.md` for how plans are loaded and enforced at runtime.

## Projects — one per customer

`plan` is a plain string (no FK — plans live in YAML, not D1).

```sql
CREATE TABLE projects (
  id          TEXT PRIMARY KEY,
  name        TEXT NOT NULL,
  owner_email TEXT NOT NULL UNIQUE,  -- Auth.js email; UNIQUE prevents duplicate provisioning
  plan        TEXT NOT NULL DEFAULT 'free'
                CHECK(plan IN ('free', 'pro', 'premium', 'enterprise')),
  status      TEXT NOT NULL DEFAULT 'active'
                CHECK(status IN ('active', 'inactive')),
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL
);
```

## SDK Keys

Server keys poll `/sdk/flags`. Client keys only call `/sdk/evaluate`. Admin keys do everything.

```sql
CREATE TABLE sdk_keys (
  id          TEXT PRIMARY KEY,
  project_id  TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  key_hash    TEXT UNIQUE NOT NULL,  -- SHA-256 hash; never store raw key
  type        TEXT NOT NULL CHECK(type IN ('server', 'client', 'admin')),
  created_at  TEXT NOT NULL,
  revoked_at  TEXT,
  expires_at  TEXT    -- NULL = no expiry (server/client keys); admin (CLI) keys expire after 90 days
);
CREATE INDEX sdk_keys_hash ON sdk_keys(key_hash);
```

## Auth Model — Stateless JWT (No Sessions Table)

Auth uses Auth.js v5 with JWT strategy (`session.maxAge = 15 * 60`). There is no
`sessions` D1 table — JWTs are stateless and self-expire after 15 minutes.

**Revocation model:** Short-lived JWT expiry replaces per-session revocation.
A compromised session self-expires within 15 minutes. If immediate global revocation
is needed, rotate `AUTH_SECRET` (both Pages and Worker env vars) — all active JWTs
are invalidated immediately.

The Worker does not verify JWTs — it has no admin endpoints. Admin auth is handled
entirely by Next.js (Auth.js session for browser, SDK key validation for CLI).

## CLI Auth Sessions

```sql
-- CLI device auth sessions. token is NEVER stored — only token_hash.
-- The raw token is returned exactly once over the poll response and immediately discarded.
-- PKCE: code_challenge = SHA256(code_verifier); verifier presented at /auth/device/complete.
CREATE TABLE cli_auth_sessions (
  state           TEXT PRIMARY KEY,              -- UUID generated by CLI
  code_challenge  TEXT NOT NULL,                 -- SHA256(code_verifier) for PKCE binding
  project_id      TEXT REFERENCES projects(id),  -- set on completion
  token_hash      TEXT,                          -- SHA-256 of minted token (FK into sdk_keys)
  status          TEXT NOT NULL DEFAULT 'pending'
                       CHECK(status IN ('pending', 'complete', 'expired')),
  created_at      TEXT NOT NULL,
  expires_at      TEXT NOT NULL,                 -- created_at + 5 minutes
  completed_at    TEXT
);
CREATE INDEX cli_auth_sessions_expires ON cli_auth_sessions(expires_at);
-- Note: raw token is generated at /auth/device/complete, returned once to CLI via /auth/device/poll,
-- then discarded. token_hash in sdk_keys is the permanent credential record.
```

## CLI Tokens

Audit trail and revocation record for CLI admin tokens. The raw token is still delivered
once via KV (one-time read in `/auth/device/poll`) but a D1 row persists for revocation
and audit. The SDK key row in `sdk_keys` is the actual credential; this table adds lifecycle
tracking on top.

```sql
CREATE TABLE cli_tokens (
  token_id    TEXT PRIMARY KEY,              -- matches sdk_keys.id for the admin key
  project_id  TEXT NOT NULL REFERENCES projects(id),
  user_email  TEXT NOT NULL,                 -- who authenticated via PKCE
  issued_at   TEXT NOT NULL,
  expires_at  TEXT NOT NULL,                 -- created_at + 90 days (matches sdk_keys.expires_at)
  revoked_at  TEXT,                          -- NULL = active; set to revoke before expiry
  last_used_at TEXT                          -- updated on each admin request (best-effort)
);
CREATE INDEX cli_tokens_project ON cli_tokens(project_id);
CREATE INDEX cli_tokens_expires ON cli_tokens(expires_at);
```

Revoking a CLI token: `PATCH /admin/keys/:id/revoke` already sets `sdk_keys.revoked_at`
(existing flow) AND must also set `cli_tokens.revoked_at` for the matching `token_id`.
The `authMiddleware` check for admin keys must verify `sdk_keys.revoked_at IS NULL` — this
is already enforced. The `cli_tokens` row provides the audit trail and the email → token
lookup needed for "revoke all tokens for user X" operations.

## System Health

## Audit Log

Every admin mutation (Server Actions and Route Handlers) writes an audit log entry via the `auditLog()` helper in `@flaglab/core`. The helper accepts the mutation context (actor, action, resource) and inserts a row. No manual audit calls needed — the `withAudit()` wrapper handles it automatically.

```sql
-- Tracks all admin mutations for compliance and debugging.
-- Written automatically by the audit middleware on every D1 write in Server Actions.
CREATE TABLE audit_log (
  id          TEXT PRIMARY KEY,
  project_id  TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  actor_email TEXT NOT NULL,               -- from Auth.js JWT or CLI token lookup
  actor_type  TEXT NOT NULL CHECK(actor_type IN ('user', 'cli', 'system')),
  action      TEXT NOT NULL,               -- e.g. 'gate.create', 'experiment.start', 'key.revoke'
  resource_type TEXT NOT NULL,             -- e.g. 'gate', 'experiment', 'sdk_key'
  resource_id TEXT,                        -- ID of affected resource (NULL for bulk ops)
  payload     TEXT,                        -- JSON: input data or diff summary
  created_at  TEXT NOT NULL
);
CREATE INDEX audit_log_project   ON audit_log(project_id, created_at);
CREATE INDEX audit_log_actor     ON audit_log(actor_email);
CREATE INDEX audit_log_resource  ON audit_log(resource_type, resource_id);
```

Retention: rows older than `audit_log_retention_days` (per plan) are purged by the daily retention cron.

## Universes

Holdout is defined HERE, not on experiments. A holdout excludes the same user
segment from ALL experiments in this universe, so the holdout group measures
cumulative impact of everything shipped.

```sql
CREATE TABLE universes (
  id            TEXT PRIMARY KEY,
  project_id    TEXT NOT NULL,
  name          TEXT NOT NULL,
  unit_type     TEXT NOT NULL DEFAULT 'user_id',
  holdout_range TEXT,              -- JSON: [lo, hi] or null — UNIVERSE-LEVEL ONLY
  created_at    TEXT NOT NULL,
  UNIQUE(project_id, name)
);
CREATE INDEX universes_project ON universes(project_id);
```

## Experiments (QE)

Groups use weights (integers summing to exactly 10000, basis points), not a 10k-element segments array.
`holdout_range` is on the universe, not here.

**Group weight invariant:** `SUM(group.weight) MUST equal exactly 10000` — enforced by the Zod schema at insert time (`z.refine(groups => groups.reduce((s,g) => s+g.weight, 0) === 10000)`). SQLite has no native array-level CHECK constraint; enforcement is application-layer only. A sum of 9999 causes 1-in-10,000 users to receive no variant (silent exclusion); 10001 causes the last group to be slightly underrepresented. Both are silent bugs — validate aggressively.

```sql
CREATE TABLE experiments (
  id              TEXT PRIMARY KEY,
  project_id      TEXT NOT NULL,
  name            TEXT NOT NULL,
  universe        TEXT NOT NULL,   -- FK → universes.name (same project)
  targeting_gate  TEXT,
  allocation_pct  INTEGER NOT NULL DEFAULT 0,  -- basis points 0–10000; 1000 = 10%
  salt            TEXT NOT NULL,
  params          TEXT NOT NULL,   -- JSON: {name: type}
  groups          TEXT NOT NULL,   -- JSON: [{name, weight, params}] weights are integers summing to exactly 10000
  status          TEXT NOT NULL DEFAULT 'draft',  -- draft|running|stopped|archived
                    CHECK(status IN ('draft', 'running', 'stopped', 'archived')),
  started_at      TEXT,
  stopped_at      TEXT,            -- set once when status transitions to 'stopped'; never overwritten
  significance_threshold REAL NOT NULL DEFAULT 0.05,  -- per-experiment configurable, not hardcoded
  sequential_testing INTEGER NOT NULL DEFAULT 0,       -- 1 = use mSPRT for always-valid sequential testing (Premium/Enterprise)
  min_runtime_days       INTEGER NOT NULL DEFAULT 0,   -- minimum days before verdict is emitted
  min_sample_size        INTEGER NOT NULL DEFAULT 100, -- minimum users per group before verdict
  cuped_frozen_at        TEXT,                          -- timestamp when CUPED baselines were frozen (at start)
  hash_version    INTEGER NOT NULL DEFAULT 1,           -- MurmurHash3 impl version; bump only on bug fix (see 04-evaluation.md)
                                                        -- In-flight experiments MUST keep their original hash_version;
                                                        -- changing it mid-run reassigns all users and invalidates results.
  updated_at      TEXT NOT NULL,
  UNIQUE(project_id, name)
);
CREATE INDEX experiments_project ON experiments(project_id);
```

## Metrics Library

Reusable analytical definitions referencing registered events.
`event_name` must match `events.name` for the same project.
`value_path` = name of a numeric event property (null for count_users/count_events).

```sql
CREATE TABLE metrics (
  id                    TEXT PRIMARY KEY,
  project_id            TEXT NOT NULL,
  name                  TEXT NOT NULL,
  event_name            TEXT NOT NULL,
  value_path            TEXT,
  aggregation           TEXT NOT NULL DEFAULT 'count_users',  -- count_users|count_events|sum|avg|retention_Nd
  winsorize_pct         INTEGER NOT NULL DEFAULT 99,  -- integer percentile 1–99
  min_detectable_effect REAL,   -- NULL = no MDE threshold; guardrail never fires below this absolute delta
  updated_at            TEXT NOT NULL,
  UNIQUE(project_id, name)
);
```

## Experiment ↔ Metrics Join

```sql
CREATE TABLE experiment_metrics (
  id            TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(8)))),
  experiment_id TEXT NOT NULL REFERENCES experiments(id) ON DELETE CASCADE,
  metric_id     TEXT NOT NULL REFERENCES metrics(id)     ON DELETE CASCADE,
  role          TEXT NOT NULL CHECK(role IN ('goal','guardrail','secondary')),
  created_at    INTEGER NOT NULL DEFAULT (unixepoch()),
  UNIQUE(experiment_id, metric_id)
);
```

Roles:

- `goal` — must improve significantly to ship
- `guardrail` — must NOT regress; any regression blocks shipping
- `secondary` — informational only, doesn't affect verdict

## Analysis Results

Written by cron (daily), read by UI.

```sql
CREATE TABLE experiment_results (
  project_id   TEXT NOT NULL,
  experiment   TEXT NOT NULL,
  metric       TEXT NOT NULL,
  group_name   TEXT NOT NULL,
  ds           TEXT NOT NULL,
  n            INTEGER,
  mean         REAL,
  variance     REAL,     -- sample variance, Bessel-corrected (÷n-1 not ÷n)
  delta        REAL,
  delta_pct    REAL,
  ci_95_low    REAL,
  ci_95_high   REAL,
  ci_99_low    REAL,
  ci_99_high   REAL,
  p_value      REAL,
  -- SRM (Sample Ratio Mismatch) detection
  expected_n      INTEGER,     -- expected users based on group weight × total exposed
  srm_p_value     REAL,        -- chi-squared p-value; < 0.01 = SRM detected
  srm_detected    INTEGER NOT NULL DEFAULT 0,  -- 1 if SRM detected; verdict suppressed when true
  -- mSPRT sequential testing (Premium/Enterprise — sequential_testing=true)
  msprt_lambda    REAL,    -- mixture likelihood ratio (NULL for non-sequential experiments)
  msprt_significant INTEGER, -- 1 if lambda > 1/alpha (NULL for non-sequential)
  -- Analysis metadata
  is_final        INTEGER NOT NULL DEFAULT 0,  -- 1 if this is the terminal result written at experiment stop
  peek_warning    INTEGER NOT NULL DEFAULT 0,  -- 1 if min_runtime_days not yet reached
  PRIMARY KEY (project_id, experiment, metric, group_name, ds)
);

-- Required for retention purge query performance
CREATE INDEX experiment_results_ds ON experiment_results(project_id, ds);
```

## CUPED Baselines

14-day pre-experiment average per user per metric. Frozen once at experiment start by `freezeCupedBaselines()`. Never updated during the experiment — rolling updates would violate CUPED's independence assumption.
Used by CUPED to reduce variance in the t-test.

```sql
CREATE TABLE user_metric_baseline (
  project_id  TEXT NOT NULL,
  user_id     TEXT NOT NULL,
  metric_name TEXT NOT NULL,
  avg_value   REAL NOT NULL,
  updated_ds  TEXT NOT NULL,
  PRIMARY KEY (project_id, user_id, metric_name)
);
```

## User Attribute Catalog

Attributes must be registered before they can be used in gate targeting rules.
The rule builder in the UI only shows registered attributes, preventing rules
that reference undefined fields.

```sql
CREATE TABLE user_attributes (
  id           TEXT PRIMARY KEY,
  project_id   TEXT NOT NULL,
  name         TEXT NOT NULL,
  type         TEXT NOT NULL
                 CHECK(type IN ('string','number','boolean','enum','date')),
  enum_values  TEXT,                   -- JSON array: ["free","pro","premium"] (enum only)
  required     INTEGER NOT NULL DEFAULT 0,  -- 1 = SDK must always include this attribute
  description  TEXT,
  sdk_path     TEXT,                   -- hint for developers: 'user.subscription.plan'
  created_at   TEXT NOT NULL,
  UNIQUE(project_id, name)
);
CREATE INDEX user_attributes_project ON user_attributes(project_id);
```

### Seed attributes on project creation

These are created automatically when a project is provisioned:

```sql
-- Seeded in Auth.js jwt callback (inline provisioning) after project INSERT
INSERT INTO user_attributes (id, project_id, name, type, required, description)
VALUES
  (uuid(), ?, 'user_id',      'string',  1, 'Unique user identifier — always required'),
  (uuid(), ?, 'anonymous_id', 'string',  0, 'Pre-auth session ID for anonymous users'),
  (uuid(), ?, 'email',        'string',  0, 'User email address'),
  (uuid(), ?, 'plan',         'enum',    0, 'Subscription plan'),
  (uuid(), ?, 'country',      'string',  0, 'ISO 3166-1 alpha-2 country code'),
  (uuid(), ?, 'company_size', 'number',  0, 'Number of employees in the organization');
```

### Three-layer enforcement

```
Layer 1 — Catalog:    Attribute registered with type + enum_values
Layer 2 — Rule UI:    Dropdown only shows registered attributes; blocks unknown ones
Layer 3 — Evaluate:   Missing attribute → rule fails safe (false) + warning in response
```

Missing attributes return `false` safely and are surfaced in the evaluate response as `_attribute_warnings` — see `03-worker-endpoints.md` § "/sdk/evaluate" for the response shape.

## Resource Limits

Plan-gated hard limits enforced at mutation time via `checkLimit()` from `@flaglab/core` — called by Next.js Server Actions before every admin INSERT. `-1` in `plans.yaml` means unlimited per plan, subject to platform ceilings (e.g. 40K gates max to stay under KV's 25MB limit). Event rate limits use Cloudflare Rate Limiting rules keyed on `project_id`, configured in the dashboard.
