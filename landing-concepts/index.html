shipeasy · AI-first landing design review 5 concepts · 4 locales · 1 pitch

Five landings. One pitch:
don't just let AI ship code — let it ship experiments.

Every concept leads with the same story: shipeasy gives your Claude/Cursor agent an MCP server, 12 Claude skills, and a CLI to create flags, run A/B tests, roll out translations, and read statistically defensible verdicts — without a human babysitting a dashboard. The variation is in voice and audience: dev-first, data-first, editorial, solo/vibe, or enterprise.

Toggle locales in-page via the picker (top-right). Press ⌘/Ctrl + Shift + K to overlay translation keys.

i18n / string-manager is an agent surface, not just a file

The string manager is a first-class primitive — same rails as flags and experiments. That means a Claude agent can call update_translation to ship a copy change, or create_experiment on a translation key to A/B test one locale without touching the others. These concepts demonstrate the client side of that story via a tiny runtime in _shared/i18n.js:

  • Locale switching without reload — ready for Next.js App Router + middleware matching
  • Interpolation: "Welcome, {{name}}" via data-i18n-vars
  • Plurals: { one, other } keyed by data-i18n-plural="5"
  • Attribute translation: data-i18n-attr="placeholder:form.email"
  • Dev-only key overlay (⌘/Ctrl+Shift+K) for designer and agent QA of coverage
  • Graceful fallback to en when a key is missing in the active locale
