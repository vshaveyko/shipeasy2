# LLM Onboarding: String Manager (i18n)

> **For AI agents.** This guide is designed to be followed autonomously.
> There is exactly **one mandatory stop**: the human must click the browser link that opens during `shipeasy auth login`.
> Every other step runs without user interaction.

---

## Step 1 — Install the CLI and MCP server

Install the ShipEasy CLI globally and register the MCP server with your AI assistant config.

```bash
npm install -g @shipeasy/cli
```

Add the MCP server to your AI assistant. For **Claude Code**, edit (or create) `~/.claude/settings.json`:

```json
{
  "mcpServers": {
    "shipeasy": {
      "command": "npx",
      "args": ["-y", "@shipeasy/mcp@latest"]
    }
  }
}
```

For **Cursor**, write the same block to `.cursor/mcp.json` in the project root.
For **Windsurf**, write it to `.windsurf/mcp.json`.
For any other assistant that supports MCP, write it to `mcp.json` in the project root.

Alternatively, one command writes all configs automatically:

```bash
shipeasy mcp install
```

Add the CLAUDE.md skill block below to your project's `CLAUDE.md` (create one at the repo root if it doesn't exist):

```markdown
## ShipEasy String Manager

This project uses ShipEasy String Manager for localization.
MCP server: `@shipeasy/mcp` — tools are prefixed `i18n_*`.

Workflow for i18n changes:

1. Call `detect_project` to confirm framework and SDK state.
2. Call `auth_check` — if not authenticated, run `shipeasy auth login` in the terminal.
   It opens a browser automatically; wait for it to print success or failure, then continue.
   Credentials are stored locally and picked up automatically by all future CLI and MCP calls.
3. Use `i18n_codemod_preview` to show diffs before writing. Call `i18n_codemod_apply` only after the human approves.
4. Use `i18n_push_keys` to sync keys to the dashboard after any codemod.
5. Use `i18n_publish_profile` to push changes to the CDN after reviewing drafts.

Never skip the preview step. Never run `i18n_codemod_apply` without `confirm: true` and explicit approval.
```

---

## Step 2 — Detect the project and install the SDK

Call `detect_project` to identify the framework, package manager, and whether the SDK is already installed:

```
mcp: detect_project {}
```

If `detect_project` returns `shipeasy.i18n_sdk.installed: false`, install the SDK for the detected language using the command returned by `detect_project` (each language SDK ships its own install instructions and post-install setup). If the SDK is already installed, skip to Step 3.

---

## Step 3 — Authenticate ⏸ STOP FOR BROWSER CLICK

Run the auth status check:

```
mcp: auth_check {}
```

If it returns `{ "authenticated": false }`, run `shipeasy auth login` in the terminal:

```bash
shipeasy auth login
```

This command opens the ShipEasy login page in your default browser automatically. The human must sign in (GitHub, Google, or magic link). When the browser flow completes, the command prints either `Authenticated successfully` or an error message and exits.

Credentials are stored in `~/.shipeasy/credentials` and are picked up automatically by all future CLI commands and MCP tool calls — no environment variables needed.

Call `auth_check` again to verify `authenticated: true`, then continue.

---

## Step 4 — Add the loader script tag

The loader script delivers published string bundles to the browser with zero JS dependencies and infinite CDN caching.

Call `i18n_install_loader` to generate the script tag parameters (creates a client SDK key automatically if needed):

```
mcp: i18n_install_loader {
  "profile": "en:prod"
}
```

The tool returns `{ data_key, profile, loader_url, script_tag, next_step }`.

Then run the CLI to inject the tag into the right file automatically:

```bash
shipeasy i18n install-loader --data-key <data_key> --profile en:prod
```

The CLI auto-detects your framework:

- **Next.js App Router** → patches `src/app/layout.tsx` inside `<head>`
- **Next.js Pages Router** → patches `pages/_document.tsx`
- **React + Vite** → patches `index.html`
- **Unknown framework** → prints the tag; paste it manually inside `<head>`

Pass `--path <file>` to override the target file. Pass `--print` to print the tag without writing files.

If `detect_project` returned `shipeasy.loader_script_tag.present: true`, skip this step.

---

## Step 5 — Create the source profile

A **profile** is a locale + environment pair (e.g. `en:prod` for English production). Create one profile that matches the language already used in the codebase:

```
mcp: i18n_create_profile {
  "name": "en:prod",
  "locale": "en",
  "environment": "prod"
}
```

If the codebase is not in English, substitute the correct BCP-47 locale code (e.g. `fr:prod`, `de:prod`).

If `list_resources { "kind": "profiles" }` already shows a matching profile, skip this step.

---

## Step 6 — Run the codemod (preview first)

The codemod is provided by the installed language SDK — `i18n_codemod_preview` and `i18n_codemod_apply` delegate to it automatically based on the detected framework.

First, preview without writing:

```
mcp: i18n_codemod_preview {
  "files": ["src/"],
  "profile": "en:prod"
}
```

Review the returned diffs. If they look correct, apply:

```
mcp: i18n_codemod_apply {
  "files": ["src/"],
  "profile": "en:prod",
  "confirm": true
}
```

The apply tool also writes `i18n-codemod-review.json` at the project root listing every string that needs manual review (dynamic strings, rich HTML, plurals, etc.).

If the project is large (>300 files), run the codemod one directory at a time to keep diffs reviewable.

---

## Step 7 — Push keys to the dashboard

After the codemod writes files, push all discovered keys to the source profile:

```
mcp: i18n_push_keys {
  "profile": "en:prod",
  "source": "codemod"
}
```

The tool returns `{ pushed_count, skipped_count, failed_keys }`. If `failed_keys` is non-empty, inspect the errors — usually malformed key names.

Verify keys arrived:

```
mcp: list_resources { "kind": "keys", "search": "" }
```

---

## Step 8 — Second pass and key descriptions

Run a second scan to catch strings the first pass missed:

```
mcp: i18n_scan_code {
  "path": "src/",
  "include_dynamic": false
}
```

For each candidate string not yet in the profile, either call `i18n_codemod_apply` on the containing file, or wrap it manually if the codemod can't handle it (dynamic key, rich HTML, etc.).

**Add descriptions to every pushed key.** For every key returned by `list_resources { "kind": "keys" }`, call:

```
mcp: i18n_create_key {
  "profile": "en:prod",
  "key": "<key_name>",
  "value": "<source string>",
  "description": "<one sentence: where this string appears and what it does>"
}
```

Good descriptions: `"CTA button on the checkout summary page — confirms the purchase"`,
`"Empty-state heading shown when the user has no active projects"`.

---

## Step 9 — Validate and publish

Validate that every key referenced in code exists in the profile:

```
mcp: i18n_validate_keys {
  "profile": "en:prod"
}
```

Fix any missing keys before proceeding.

Publish the source profile to the CDN:

```
mcp: i18n_publish_profile {
  "profile": "en:prod"
}
```

Strings are live at the edge within seconds.

---

## What's next

- **Add translations** — create target profiles (`fr:prod`, `de:prod`, etc.) and run `i18n_translate_draft` using the Anthropic API from your machine.
- **Review drafts** — open **String Manager → Drafts** in the dashboard to approve or edit AI-generated translations before publishing.
- **Set up validation in CI** — add `shipeasy i18n validate` as a pre-merge check to prevent missing keys from reaching production.
- **Automate with the `translate_site` prompt** — if your site has the `/.well-known/i18n.json` manifest, the AI can discover the full state and manage ongoing translation automatically.

See the [String Manager docs](/i18n) for the full reference.
