ShipEasy ships every product behind a typed **MCP server** (`shipeasy-mcp`). The CLI installs it into your AI assistant's config, the agent discovers the tools, and these guides tell the agent the right sequence of calls to make for a given workflow. You answer one browser auth prompt; the agent does the rest.

## How it works [#how-it-works]

<Steps>
  <Step title="Install the MCP server">
    The CLI ships a one-shot installer that writes the right MCP config for whichever assistants it finds on your machine — `~/.claude/settings.json`, `.cursor/mcp.json`, `.windsurf/mcp.json`.

    ```bash
    npm install -g @shipeasy/cli
    shipeasy mcp install
    ```

    Under the hood it registers `npx -y @shipeasy/mcp@latest` as the `shipeasy` MCP server. The binary is `shipeasy-mcp`; it runs locally over stdio, never as a hosted endpoint.

  </Step>

  <Step title="Log in once">
    ```bash
    shipeasy login
    ```

    The CLI runs an OAuth device-code flow against the ShipEasy worker (`/auth/device/*`) and stores a long-lived credential at `~/.shipeasy/credentials`. The MCP server reads from the same file. **The agent never sees your token** — every tool call attaches the credential server-side.

  </Step>

  <Step title="Hand the agent a guide URL">
    Open your AI assistant in your project and say something like:

    > _"Follow https://docs.shipeasy.ai/llms/i18n-strings to onboard this project to ShipEasy."_

    The agent fetches the page, reads the steps, and starts calling the appropriate `mcp__shipeasy__*` tools in order. It pauses for human input only at the points the runbook marks with <code>STOP</code> (typically: browser auth, ambiguous codemod cases).

  </Step>
</Steps>

## What the MCP server exposes

The `shipeasy-mcp` binary advertises a typed tool surface grouped by product. Every guide on this site is a recipe for chaining these tools:

<ApiTable>
  <ApiRow
    name="auth_check"
    type="tool"
    desc={
      <>
        Returns <code>{`{ authenticated: boolean, user, project }`}</code>. First call in every
        workflow — if it returns <code>false</code>, the agent must ask the human to run{" "}
        <code>shipeasy login</code>.
      </>
    }
  />
  <ApiRow
    name="auth_login / auth_logout"
    type="tool"
    desc="Wrap the device-code flow. Login opens a browser; the human clicks once. The agent waits for the tool to return."
  />
  <ApiRow
    name="detect_project"
    type="tool"
    desc={
      <>
        Inspects the working directory and returns <code>frameworks</code>,{" "}
        <code>entry_points</code>, and <code>shipeasy.{`{i18n_sdk, loader_script_tag}`}</code>. Lets
        the agent skip framework guesswork.
      </>
    }
  />
  <ApiRow
    name="list_resources / get_resource"
    type="tool"
    desc={
      <>
        Read-only views into the project — list <code>profiles</code>, <code>keys</code>,{" "}
        <code>gates</code>, <code>configs</code>, <code>experiments</code>. Cheap to call; the agent
        uses these to verify state between mutations.
      </>
    }
  />
  <ApiRow
    name="i18n_*"
    type="11 tools"
    desc={
      <>
        Polylang surface — <code>install_loader</code>, <code>create_profile</code>,{" "}
        <code>scan_code</code>, <code>codemod_preview</code>, <code>codemod_apply</code>,{" "}
        <code>push_keys</code>, <code>create_key</code>, <code>translate_draft</code>,{" "}
        <code>validate_keys</code>, <code>publish_profile</code>, <code>discover_site</code>.
      </>
    }
  />
  <ApiRow
    name="exp_*"
    type="6 tools"
    desc={
      <>
        Configs / experiments surface — <code>create_gate</code>, <code>create_config</code>,{" "}
        <code>create_experiment</code>, <code>start_experiment</code>, <code>stop_experiment</code>,{" "}
        <code>experiment_status</code>.
      </>
    }
  />
  <ApiRow
    name="get_sdk_snippet"
    type="tool"
    desc="Returns ready-to-paste init code for the detected framework, scoped to the current project's API key."
  />
</ApiTable>

<Callout type="info" title="No tool runs your shell">
  MCP tools mutate ShipEasy state through the public API. Anything that touches your filesystem (the
  codemod, the loader install) runs the local CLI binary, which prints what it&apos;s about to do
  and respects <code>--dry-run</code>. There is no <code>exec_arbitrary_command</code> tool.
</Callout>

## Write your own guide

The guides are plain MDX. If your team has conventions — a specific source root, a review step, a Slack channel that needs pinging on publish — fork one, edit it, host it. The agent doesn&apos;t care where the URL points; it cares that the page describes a sequence of MCP tool calls.

<Steps>
  <Step title="Start from an existing guide">
    Copy `content/docs/llms/i18n-strings.mdx` from [the docs
    repo](https://github.com/shipeasy-ai/shipeasy2/tree/main/apps/docs/content/docs/llms). It&apos;s
    the canonical pattern: numbered steps, explicit tool names, explicit JSON args, explicit failure
    handling.
  </Step>
  <Step title="Mark every human stop">
    Use the literal token <code>STOP</code> in step titles where the agent must wait — browser auth,
    codemod approval, anything that mutates production. Agents are trained to honour this.
  </Step>
  <Step title="Reference tools by their MCP name">
    Use the canonical tool name (<code>i18n_codemod_apply</code>, not <em>"the apply step"</em>).
    Show the full JSON arg shape. Agents pattern-match on these.
  </Step>
  <Step title="Host it on a stable URL">
    Anywhere the agent can fetch — your docs site, a GitHub raw URL, a gist. Hand the URL to your
    team and to your agents.
  </Step>
</Steps>

<Callout type="warn" title="Don't hardcode runbooks into your codebase">
  Resist the urge to paste a guide into `CLAUDE.md` or your project&apos;s `AGENTS.md`. The MCP tool
  surface evolves; the URL stays current; a copy goes stale immediately. Reference the URL,
  don&apos;t inline it.
</Callout>
