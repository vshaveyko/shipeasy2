title: Quickstart

This walks through the full flow end to end: create a gate, read it from a server, roll it out, kill-switch it.

## Prerequisites

- A ShipEasy project. Sign up at [shipeasy.ai](https://shipeasy.ai) — the free tier is enough.
- A server SDK key. Find it in **Project → SDK keys → Create → server**, or `shipeasy keys create`.
- The SDK installed: `npm install @shipeasy/sdk`.

<Steps>
  <Step title="Initialise the SDK">
    Initialise once at boot. The server SDK polls flag updates in the background — no per-request cost.

    ```ts title="src/lib/shipeasy.ts"
    import { configureShipeasy, flags } from "@shipeasy/sdk/server";

    configureShipeasy({
      apiKey: process.env.SHIPEASY_SERVER_KEY!,
      env: process.env.NODE_ENV === "production" ? "prod" : "dev",
    });

    export const ready = flags.init();
    ```

    Import this once from your entrypoint and `await ready` before serving requests.

  </Step>

  <Step title="Create a gate">
    From the CLI:

    ```bash
    shipeasy flags create new-checkout-flow --rollout 0
    ```

    Or in the dashboard: **Configs → Gates → New gate**, name `new-checkout-flow`, leave the rollout at `0%`.

  </Step>

  <Step title="Gate your code">
    ```ts
    import { flags } from "@shipeasy/sdk/server";

    export async function getCheckoutComponent(user) {
      if (flags.gate("new-checkout-flow", user)) {
        return import("./checkout/new");
      }
      return import("./checkout/legacy");
    }
    ```

    `gate()` is synchronous and reads from the in-process cache. Pass any user attributes you want available to targeting rules — at minimum `{ user_id }`.

  </Step>

<Step title="Test locally">
  Add yourself to the **overrides** list in the dashboard (or pass
  `?shipeasy_gate_new-checkout-flow=true` if you have the [browser DevTools
  overlay](/get-started/sdks#devtools) loaded). The flag returns `true` for you and `false` for
  everyone else.
</Step>

  <Step title="Roll it out">
    Bump the rollout from `0%` → `5%` → `25%` → `100%` over a few hours, watching your dashboards. Bucketing is deterministic by user ID, so the same 5% always sees the new path.

    ```bash
    shipeasy flags create new-checkout-flow --rollout 25
    ```

  </Step>

  <Step title="Killswitch">
    If something goes wrong, flip the killswitch. It overrides every targeting rule and serves `false` everywhere within the SDK&apos;s next poll.

    Dashboard: toggle the red **Killswitch** chip. CLI: `shipeasy flags disable new-checkout-flow`.

  </Step>
</Steps>

## What just happened

When you flipped that rollout from `25` to `100`, the dashboard:

1. Updated the project&apos;s flag row in D1.
2. Rebuilt the project&apos;s `:flags` blob in Cloudflare KV.
3. Purged the affected URLs from the CDN.
4. Within the SDK&apos;s poll interval (default 30s, faster on Pro), every server worldwide saw the new value.

No deploy. No redeploy. No restart. No backend round-trip on the read.
