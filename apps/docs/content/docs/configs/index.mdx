ShipEasy **Configs** gives you two primitives:

- **Gates** — boolean feature flags with optional targeting rules and percentage rollouts.
- **Dynamic values** — typed key/value configs (string, number, boolean, JSON) you can change without shipping code.

Both are stored in Cloudflare KV with infinite CDN TTL and surgically purged on change. A flag flip you make in the dashboard is visible to your SDK in seconds, worldwide, with **no per-request network call** from your code.

## The 5-minute path

<Quickstart title="Your first gate" time="~5 minutes">
  <QuickstartStep
    num={1}
    label="CREATE"
    title="Create the gate"
    cmd="shipeasy gate create new-checkout-flow"
  >
    Default off. Targeting and rollout % are optional — a gate with neither is a kill-switch.
  </QuickstartStep>
  <QuickstartStep
    num={2}
    label="WIRE"
    title="Read it in code"
    cmd={`flags.gate("new-checkout-flow", ctx)`}
  >
    The SDK keeps the bundle in process; evaluation is local and synchronous.
  </QuickstartStep>
  <QuickstartStep
    num={3}
    label="ROLL OUT"
    title="Bump the percentage"
    cmd="shipeasy gate rollout new-checkout-flow --percent 25"
  >
    Watch your dashboards. Bump again. Or kill it instantly with <code>--percent 0</code>.
  </QuickstartStep>
</Quickstart>

## Wire the SDK

<InstallTabs npm="npm install @shipeasy/sdk" />

```ts
import { configureShipeasy, flags } from "@shipeasy/sdk/server";

configureShipeasy({ apiKey: process.env.SHIPEASY_SERVER_KEY!, env: "prod" });
await flags.init();

if (flags.gate("new-checkout-flow", { user_id })) {
  // new path
} else {
  // old path
}
```

## API · `flags.gate`

  <ApiRow

      </>
    }
  />
  <ApiRow
    name="defaultValue"
    optional
    type="boolean"
    desc="Returned when the gate is missing from the bundle (e.g. SDK not initialised). Defaults to false — fail closed."
  />

The return is a plain `boolean`. There is no async, no Promise, no fetch — the bundle is in memory.

## Why edge KV

<Steps>
  <Step title="Reads are free">
    The SDK polls a KV blob in the background and evaluates locally. No round-trip per request, no
    rate limits, no per-evaluation cost.
  </Step>
  <Step title="Writes are surgical">
    Flipping a gate in the dashboard rebuilds one KV blob (per project, per env) and explicitly
    purges the CDN URL. The next request anywhere on Earth sees the new value.
  </Step>
  <Step title="Plan-driven poll interval">
    Free tier polls every 60s. Pro tier every 10s. Enterprise: streaming push. The SDK handles the
    difference; your code is identical.
  </Step>
</Steps>

<Callout type="info" title="Configs and Polylang share KV">
  Both products write to the same Cloudflare KV namespace (scoped per project). That&apos;s why a
  translation publish and a gate flip both go live in seconds — the same edge plumbing, the same
  purge mechanism.
</Callout>

## Gate vs. dynamic value — pick one

| Question                            | Gate | Dynamic value |
| ----------------------------------- | ---- | ------------- |
| Is the answer boolean?              | ✓    | –             |
| Does the answer depend on the user? | ✓    | –             |
| Do you need a typed JSON payload?   | –    | ✓             |
| Do you want global rollout %?       | ✓    | –             |
| Is this configuration vs. behavior? | –    | ✓             |

A useful rule: if you&apos;d say _"show feature X"_, gate it. If you&apos;d say _"set feature X to value Y"_, use a dynamic value.
