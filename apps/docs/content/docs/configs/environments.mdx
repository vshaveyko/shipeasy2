Every project has three environments: **dev**, **staging**, and **prod**. They&apos;re not separate projects — they&apos;re **slices of the same project** so you don&apos;t maintain three parallel lists of flags. Each gate and value holds an independent record per environment, the SDK pins to one environment at boot, and you promote between them with a click.

This page covers how to wire the environment in your SDK, how promotion works, and the operational rules around per-environment SDK keys.

## Choosing an environment in the SDK

Server:

```ts
import { configureShipeasy } from "@shipeasy/sdk/server";

configureShipeasy({
  apiKey: process.env.SHIPEASY_SERVER_KEY!,
  env: process.env.NODE_ENV === "production" ? "prod" : "dev",
});
```

Browser:

```ts
import { configureShipeasy } from "@shipeasy/sdk/client";

configureShipeasy({
  apiKey: import.meta.env.VITE_SHIPEASY_CLIENT_KEY,
  env: import.meta.env.MODE === "production" ? "prod" : "dev",
});
```

The SDK fetches the bundle for that environment only. There&apos;s no per-call switching — pick once at boot. (If you genuinely need to evaluate against another environment from a single process — say, a CI job validating prod — instantiate a second client with a different `apiKey` + `env`.)

## Per-environment values [#dev]

A flag like `new-checkout-flow` has three independent records:

|           | enabled | rollout | killswitch |
| --------- | ------- | ------- | ---------- |
| `dev`     | true    | 100%    | off        |
| `staging` | true    | 100%    | off        |
| `prod`    | true    | 5%      | off        |

Changing one row never affects the others. The same is true for [dynamic values](/configs/values):

```ts
// dev    → "https://api-dev.example.com"
// stage  → "https://api-staging.example.com"
// prod   → "https://api.example.com"
const apiBase = flags.value<string>("api-base-url");
```

This is why you&apos;d use a dynamic value over an env var: an env var requires a redeploy to change, the value flips in &lt; a minute and rolls back in one click.

## Promoting changes [#staging]

The dashboard&apos;s **Promote** action copies a flag&apos;s configuration from one environment to another in one click. The promotion preserves overrides if both environments allow the same user IDs (they do by default).

You can also promote from the CLI:

```bash
shipeasy flags promote new-checkout-flow --from staging --to prod
```

A diff is shown and you confirm:

```text
- prod  enabled=false rollout=0%   killswitch=off  rules=[]
+ prod  enabled=true  rollout=5%   killswitch=off  rules=[plan eq pro]
       overrides:    +u_qa_alice   +u_qa_bob
Continue? [y/N]
```

Promotion does **not** auto-flip the killswitch — production keeps its own emergency control, independent from staging.

```bash
# Promote a value, scoped to a single key
shipeasy values promote api-base-url --from staging --to prod

# Dry run to preview the diff without writing
shipeasy flags promote new-checkout-flow --from staging --to prod --dry-run
```

<Callout type="success" title="Promotion is just a write">
  Promotion isn&apos;t a special-case API — it&apos;s a regular write to the target environment,
  with the source environment&apos;s values prefilled. Every promotion shows up in the audit log of
  the destination env, attributed to the user who clicked the button.
</Callout>

## Per-environment SDK keys [#prod]

Each SDK key is **scoped** to one environment. A `prod` server key cannot read `dev` flags and vice versa. This is enforced at the edge — there&apos;s no client-side check to bypass.

- Generate keys in **Project &rarr; SDK keys** with kind `server` / `client` and env `dev` / `staging` / `prod`.
- Store one set per deploy environment in your env vars / secret manager.
- Rotate independently — a leaked dev key is not a prod incident.

```bash
shipeasy keys create --kind server --env prod   # writes a key + prints once
shipeasy keys create --kind client --env dev
shipeasy keys list                              # all keys, scoped, redacted
shipeasy keys revoke <id>                       # immediately invalidated at edge
```

Server keys can read everything in their env. Client keys can only read **published** flags and values (no drafts, no overrides metadata) — safe to embed in browser bundles.

## Environments and experiments

[Experiments](/experiments) are **environment-scoped**. An experiment running in `prod` does not assign or collect data in `dev`. This means:

- You can stage an experiment&apos;s config in dev/staging without affecting prod metrics.
- Test SDK integration locally with `env: "dev"` and the experiment will assign you to a variant the same way prod will — but the exposure events go to the dev project rows, not your real analysis.
- Promotion of an experiment from staging to prod resets exposure counters; you&apos;re starting a clean run.

## Environments vs branches

Environments are **runtime** (&quot;which deployment am I&quot;), not source-control (&quot;what code am I&quot;). Don&apos;t try to map preview deploys 1:1 to environments — they all use `dev` (or `staging` if your preview pipeline calls itself that). Per-PR isolation is best done with [overrides](/configs/gates#overrides) on a stable user attribute like `pr_number` or `branch`.

```ts
flags.gate("experimental-pricing", {
  user_id,
  branch: process.env.VERCEL_GIT_COMMIT_REF, // attribute, not env
});
```

…then add a targeting rule `branch eq feat-new-pricing` to scope the flag to that branch&apos;s preview deploys.

## CI: pin the env per pipeline

```yaml title=".github/workflows/deploy-prod.yml"
env:
  SHIPEASY_SERVER_KEY: ${{ secrets.SHIPEASY_SERVER_KEY_PROD }}
  SHIPEASY_ENV: prod

steps:
  - run: shipeasy flags validate ./src --env $SHIPEASY_ENV
  - run: pnpm build
  - run: pnpm deploy
```

The validate step catches references to gates/values that don&apos;t exist in the target env. A common cause: someone created a flag in `dev`, shipped code that reads it, and forgot to promote to `prod`. The pipeline fails, you promote, you re-run.

## Cross-env reporting

The dashboard&apos;s **Configs &rarr; Diff** view compares any two environments side-by-side, showing every gate and value where they disagree. It&apos;s the easiest way to spot drift — gates that have been at 100% in `dev` for months but never promoted, or values that quietly diverged during an incident.

```bash
shipeasy diff --from staging --to prod
```

```text
GATE   new-checkout-flow      staging=on@100%   prod=on@5%      ←  ramp pending
GATE   experimental-pricing   staging=on@100%   prod=missing    ←  not promoted
VALUE  rate-limit             staging=200       prod=100        ←  drift
```
