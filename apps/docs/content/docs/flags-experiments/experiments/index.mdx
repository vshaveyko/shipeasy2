ShipEasy **Experiments** lets you ask _"is the new thing actually better?"_ and get a defensible answer.

You define what you want to measure (a **metric**), set up variant payloads (the **groups**), let users land in one bucket or another, log conversions, and look at the dashboard the next morning. The platform handles assignment, exposure stitching, deduplication, daily aggregation, and the statistics.

## The 5-minute mental model

You ship feature X behind a flag. You believe it&apos;ll lift `purchase_conversion`. So instead of just gating it, you make it an **experiment**: half the eligible users see X, half don&apos;t. Both groups are tracked. The next day, you look at the dashboard and read either _"X lifts conversion by 3.1% (p=0.002)"_ — or _"no significant difference, scrap it."_

The honest version requires more care than that summary, and the docs walk through it.

## When to gate vs. when to experiment

| Situation                                           | Gate | Experiment |
| --------------------------------------------------- | ---- | ---------- |
| Is the change reversible?                           | ✓    | ✓          |
| Does it have a known clear win (security fix, bug)? | ✓    | –          |
| Do you need an answer to "did it work"?             | –    | ✓          |
| Cheap to ship, expensive to undo (data migration)?  | ✓    | –          |
| Affects a metric you&apos;re paid to move?          | –    | ✓          |

A practical workflow is to gate first (so the rollout is safe), then **promote the gate to an experiment** once the feature is stable.

## Wire the SDK

<InstallTabs npm="npm install @shipeasy/sdk" />

```ts
import { configureShipeasy, flags, experiments } from "@shipeasy/sdk/server";

configureShipeasy({ apiKey: process.env.SHIPEASY_SERVER_KEY!, env: "prod" });
await flags.init();

const result = experiments.assign<{ label: string }>("checkout-cta", {
  user_id,
  plan,
  country,
});
const label = result.params.label; // "Pay" or "Buy now"
```

```ts
// Wherever the purchase succeeds:
flags.track(user_id, "purchase", { value: orderTotal });
```

The first time you call `assign()` for a user, an **exposure** is logged automatically. Subsequent calls in the same process don&apos;t re-log — exposures are deduplicated.

The return shape:

```ts
{
  inExperiment: boolean,   // false if stopped, holdout, or excluded
  group: string,           // "control" | "v1" | …
  params: T,               // typed payload from the assigned group
  reason: AssignmentReason // "assigned" | "holdout" | "stopped" | "excluded"
}
```

## What the daily analysis does

<Steps>
  <Step title="Cron enqueues a job per project">
    A scheduled trigger on the Edge Worker fans out one queue message per project that has running
    experiments.
  </Step>
  <Step title="Consumer scans Analytics Engine">
    For each project, the consumer pulls yesterday&apos;s exposures and events from Analytics
    Engine, joins by <code>user_id</code>, and aggregates per metric × group.
  </Step>
  <Step title="Welch's t-test">
    Lift, two-sided p-value, 95% CI. Per metric, per group, vs. the control group.
  </Step>
  <Step title="Write back to D1">
    Results land as plain rows in D1, scoped to the project. The dashboard reads them; you can
    export them.
  </Step>
</Steps>

<Callout type="warn" title="Don't peek and stop early">
  The p-values are valid for **fixed-horizon** tests. If you sneak a look every hour and stop the
  moment something turns significant, you'll get false wins. Pre-decide the experiment&apos;s
  duration based on traffic.
</Callout>

<Callout type="warn" title="Don't change variants mid-flight">
  Changing `params` on a running experiment invalidates the analysis. Stop the experiment, create a
  new one with `v2` in the name.
</Callout>
