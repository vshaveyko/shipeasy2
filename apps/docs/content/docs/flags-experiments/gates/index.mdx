A **gate** is a single boolean answer for a single user. It is the unit of <em>&quot;should I do the new thing?&quot;</em> in your code. Gates are evaluated locally — the SDK keeps the bundle in process and the call is a hash table lookup, not a network round trip.

Use a gate when the answer is yes/no. For typed payloads (strings, numbers, JSON), reach for a [dynamic value](/configs/values). For yes/no with measurement attached, reach for an [experiment](/experiments).

## Evaluation order

For a request `gate(name, user)`:

```
1. killswitch ON                       → false
2. enabled OFF                         → false
3. user.user_id ∈ overrides[true]      → true
4. user.user_id ∈ overrides[false]     → false
5. any rule fails                      → false
6. murmur3(salt + user_id) % 10000 < rolloutPct → true
7. otherwise                           → false
```

That order matters: rules are evaluated **before** the rollout. <em>&quot;100% of `plan = pro`&quot;</em> means <em>100%</em> of pro users, not <em>the same 100% bucket</em> filtered down to pros. The rollout filter never fights a targeting rule.

<Callout type="info" title="Why deterministic bucketing matters">
  Same user, same gate, same answer — every time, on every server, in every language SDK. No
  flicker, no &quot;why does this user see different things on different requests&quot;, no need to
  persist assignments anywhere. The hash is the persistence.
</Callout>

## API

### Server SDK

```ts
import { configureShipeasy, flags } from "@shipeasy/sdk/server";

configureShipeasy({ apiKey: process.env.SHIPEASY_SERVER_KEY!, env: "prod" });
await flags.init();

const enabled = flags.gate("new-checkout-flow", {
  user_id: "u_4f2a",
  plan: "pro",
  country: "US",
});
```

### Browser SDK

```ts
import { configureShipeasy } from "@shipeasy/sdk/client";

const client = configureShipeasy({ apiKey, env: "prod" });
await client.identify({ user_id, plan, country });

if (client.getFlag("new-checkout-flow")) {
  // …
}
```

`gate()` / `getFlag()` is synchronous. If you call it before `init()` resolves, you get `false` and a dev-mode warning. Always `await flags.init()` once at boot.

The return is a plain `boolean`. There is no async, no Promise, no fetch — the bundle is in memory.

## Killswitch [#killswitch]

The killswitch is a separate field from `enabled` so you can keep targeting and rollout intact while turning the feature off. Three places to flip it:

- **Dashboard**: red **Killswitch** chip on the gate row.
- **CLI**: `shipeasy flags disable <name> --killswitch` (without `--killswitch` it toggles `enabled` instead).
- **API**: `PATCH /api/flags/<name>` with `{ "killswitch": true }`.

A killed gate returns `false` everywhere within the SDK&apos;s next poll. Re-enabling restores the full prior state — overrides, rules, rollout %, salt, all of it. Treat the killswitch as your incident lever; treat `enabled` as your &quot;feature is shipped, deactivated cleanly&quot; lever.

<Callout type="warn" title="Killswitch is a contract, not a panic button">
  A killed gate goes false within the next poll, not instantly. With the default 30s poll a
  worst-case worker still serves the new path for ~30s after you hit the switch. For sub-second
  cutoff, use the Pro 10s poll or the Enterprise streaming push.
</Callout>

## Overrides [#overrides]

In the dashboard, expand a gate and add user IDs under **Overrides &rarr; Always on** or **Always off**. Overrides:

- Bypass `enabled`, `killswitch`, all rules, and the rollout.
- Are scoped to **`user_id`** only (no attribute matching — it&apos;s a literal set lookup).
- Are great for: QA accounts, internal dogfooders, customer-success demos, repro of a bug a single customer is hitting.

```ts
// dashboard JSON shape:
{
  "name": "new-checkout-flow",
  "overrides": {
    "always_on":  ["u_qa1", "u_qa2", "u_demo"],
    "always_off": ["u_legacy_partner"]
  }
}
```

If you need a percentage-based &quot;always-on for cohort X&quot;, use a [targeting rule](/configs/targeting) on a custom attribute (`internal: true`) instead. Overrides scale linearly per-row in KV; rules are O(1).

## Naming [#enabled]

- **kebab-case**: `new-checkout-flow`, not `newCheckoutFlow`.
- **Describe the change, not the abstract feature**: `enable-redis-pool` ages better than `redis-pool` (which sounds like a permanent feature, not a temporary gate).
- **Prefix by area** for grouping: `checkout-`, `nav-`, `infra-`.
- **Avoid double-negatives**: prefer `show-banner` over `hide-banner`.

## Cleaning up [#rules]

Stale flags are technical debt — once a feature is fully launched, delete the flag and the dead code. The CLI helps:

```bash
# Every gate currently at enabled: true, rollout: 100%
shipeasy flags list --json \
  | jq '.[] | select(.enabled == 1 and .rolloutPct == 10000) | .name'
```

These are candidates for removal. Run a `shipeasy flags validate ./src` after the deletion to catch any code references that need to be ripped out.

The dashboard also surfaces a **Cleanup** view that lists gates which have been at 100% (or 0%) for more than 30 days, with a one-click PR-creation button if you&apos;ve connected a GitHub app.

## The salt [#salt]

Each gate has its own salt. If you ever want to **re-shuffle** the rollout buckets without changing the percentage (say, to avoid the same 5% always being chosen for every gate), bump the salt:

```bash
shipeasy flags update my-feature --salt v2
```

That&apos;s rare. The default salt is fine for almost every case. The salt is also why two gates at 50% don&apos;t serve the same 50% of users — different salts, different buckets, independently distributed.

## Audit log

Every write to a gate is recorded in the audit log: who, when, from what (dashboard, CLI, API), and the diff. Find it under **Configs &rarr; Gates &rarr; \<gate\> &rarr; History**. Each row links to a one-click revert.

## Limits [#rollout]

| Plan       | Gates per project | Rules per gate | Overrides per gate |
| ---------- | ----------------- | -------------- | ------------------ |
| Free       | 50                | 5              | 100                |
| Pro        | 500               | 20             | 10,000             |
| Enterprise | Unlimited         | Unlimited      | Unlimited          |

If you hit a limit, audit for stale flags first; those numbers are very generous if you delete what you ship.
