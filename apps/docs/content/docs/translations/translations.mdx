Most teams have hundreds of keys and a handful of locales. Translating that by hand is slow; outsourcing is expensive and slow. **AI translations** give you a 90% draft you can review in an hour — and the rest of this doc is about getting that 90% as high as possible.

ShipEasy Translations uses **Anthropic Claude** as the translation engine. You can use ShipEasy Translations's pooled API key (Pro plan) or bring your own (BYOK). Either way every translation lands as a [draft](/translations/drafts) — never auto-merged.

## Why a model, not a translation API

Traditional translation APIs (Google, DeepL) translate sentence-by-sentence with no context. They cannot:

- Read a description like "verb, ≤12 chars" and adjust word choice.
- See sibling keys and keep terminology consistent across a UI.
- r a glossary of brand terms that should never be translated.
- Handle ICU plural categories that don't exist in the source locale.
- Use a brand voice ("warm, technical, never marketing-speak").

A model can. The trade-off is cost (still cheap — see below) and the need to review. ShipEasy Translations's review flow ([Drafts](/translations/drafts)) makes that fast.

## Setup

<Steps>
  <Step title="Add an Anthropic API key">
    Either use ShipEasy Translations's pooled key (no setup; Pro/Enterprise plans only), or add your own:

    ```bash
    shipeasy projects set-secret ANTHROPIC_API_KEY sk-ant-...
    ```

    The key is stored encrypted on the project (AES-GCM, KMS-wrapped). It is used only for translation calls; never logged, never echoed in dashboards. Rotation: re-run the command with a new key — the old one is overwritten and any in-flight translations finish on the new key.

  </Step>

  <Step title="Pick a model">
    Default is `claude-sonnet-4-6` — fast, accurate, cost-effective. For tricky locales (Japanese, Arabic, regional Spanish, traditional Chinese), `claude-opus-4-7` produces noticeably higher-quality drafts at higher cost. For high-volume marketing copy where you'll review heavily anyway, `claude-haiku-4-5` is roughly 5× cheaper.

    ```bash
    shipeasy projects config set i18n.model claude-opus-4-7
    ```

    You can also override per-locale:

    ```bash
    shipeasy projects config set i18n.model.ja claude-opus-4-7
    shipeasy projects config set i18n.model.es claude-haiku-4-5
    ```

  </Step>

  <Step title="Set tone">
    A short brand voice guide lifts quality dramatically. Keep it tonal, not exhaustive — Claude reads this on every call, longer prompts cost more and dilute focus.

    ```bash
    shipeasy projects config set i18n.tone "Friendly, modern, slightly playful. \
      Address users with informal pronouns where the locale supports it (tu / du). \
      Avoid corporate clichés. Buttons should be verbs, never nouns."
    ```

    Per-locale tone overrides exist for cases where the global voice doesn't fit — Japanese formality (です/ます vs casual), German Sie/du, Arabic register:

    ```bash
    shipeasy projects config set i18n.tone.ja "Polite (です/ます), product-friendly, no keigo flourishes."
    shipeasy projects config set i18n.tone.de "Use du, not Sie. Tech-startup register."
    ```

  </Step>

  <Step title="Set the glossary">
    A glossary is the difference between "feature gate" being translated as _drapeau_ (literal flag) vs left as-is. Map a term to itself for do-not-translate; map to a target form for forced translation.

    ```bash
    shipeasy projects config set i18n.glossary --json '{
      "ShipEasy Translations": "ShipEasy Translations",
      "ShipEasy": "ShipEasy",
      "checkout": "checkout (do not translate to French as paiement)",
      "flag": "feature gate",
      "workspace": { "fr": "espace de travail", "de": "Arbeitsbereich" }
    }'
    ```

    Object values give per-locale overrides; string values apply to every locale.

  </Step>
</Steps>

## What Claude actually sees

For each key being translated, the prompt assembled by the worker includes:

- **System prompt**: target locale, region, brand tone (global + per-locale), instructions for ICU plural categories, length-budget rule.
- **Glossary**: only entries relevant to the current batch (matched by substring), to keep the prompt small.
- **Per-key block**: key path, source value, description (if any), max-length hint (if any), placeholders allowlist.
- **Sibling context**: up to 8 neighbouring keys from the same chunk or namespace, with their already-translated values in the target profile (when present). This is what keeps terminology consistent across a UI surface.
- **Pre-existing translation**: only when running with `--reflow`.

A representative prompt structure:

```
SYSTEM:
You are translating UI strings from en → fr.
Region: France. Tone: friendly, informal (tu).
Preserve {{placeholders}} verbatim. Preserve ICU plural structure.
Use the glossary for these terms: ...

USER:
[ key: checkout.summary.cta_primary ]
source: "Pay"
description: "CTA button on checkout summary; verb; ≤12 chars"
max_length: 12
siblings (already translated):
  - checkout.summary.heading: "Récapitulatif"
  - checkout.summary.subtotal: "Sous-total"
  - checkout.summary.tax: "TVA"
Return only the translated value, no quotes, no commentary.
```

Output is parsed and validated (placeholders match, length within 1.3× of `max_length`, plural categories present). Failures are retried once with a stricter prompt; persistent failures are flagged for the reviewer with a yellow warning rather than dropped silently.

## Running a translation

```bash
shipeasy i18n translate --profile fr --ai
```

By default it translates only **missing** keys — keys present in the source profile but absent (or empty) in the target. Useful flags:

| Flag              | Effect                                                                          |
| ----------------- | ------------------------------------------------------------------------------- |
| `--stale-only`    | Re-translate only keys whose source value changed since the last translation.   |
| `--reflow`        | Revisit existing values for tone/glossary consistency. Generates a lot of diff. |
| `--chunk <name>`  | Limit to a single chunk. Faster and cheaper.                                    |
| `--key <pattern>` | Glob filter: `--key 'checkout.*'`.                                              |
| `--model <id>`    | One-shot model override (doesn't change project default).                       |
| `--ci`            | Non-interactive; non-zero exit if the draft is empty.                           |
| `--dry-run`       | Build the prompts and estimate cost; do not call the API.                       |

The CLI streams progress; long runs are resumable. If a run is interrupted, the next invocation picks up at the last completed key — drafts are written to D1 in batches of 25 so you never lose more than that.

## Cost and rate

Translations are billed per token, with the project's key (or the pooled key for Pro). Rough order of magnitude:

| Keys   | `claude-haiku-4-5` | `claude-sonnet-4-6` | `claude-opus-4-7` |
| ------ | ------------------ | ------------------- | ----------------- |
| 100    | ~$0.02             | ~$0.10              | ~$0.50            |
| 1,000  | ~$0.20             | ~$1                 | ~$5               |
| 10,000 | ~$2                | ~$10                | ~$50              |

Throughput is around 5–20 keys/second depending on context length and model. The CLI streams progress; long runs are resumable. For BYOK on Pro/Enterprise plans, prompt caching is enabled by default — system prompt and glossary cache for 5 minutes, dropping cost by ~30% on multi-locale runs.

<Callout type="info" title="Hosted vs BYOK">
  Hosted is simpler — pay ShipEasy Translations per word translated, no model decisions. BYOK is cheaper at scale
  and gives you direct visibility in your Anthropic console. Both run identical prompts with
  identical retention (zero — ShipEasy Translations sets <code>data_retention: 0</code> on every Anthropic call).
</Callout>

## Streaming and observability

The dashboard's translation page streams keys as they complete. You see:

- A live progress bar (translated / total).
- Current cost (estimated from token counts as they stream).
- ETA, recomputed every ten seconds from observed throughput.
- A per-key feed: source value → proposed translation, scrollable.
- Errors and retries inline.

The same data is available over the API:

```bash
curl -N https://api.shipeasy.ai/translations/translate/jobs/job_01HZ.../events \
  -H "Authorization: Bearer $TOKEN"
```

Returns a Server-Sent Events stream of `progress`, `key.translated`, `key.failed`, and `done` events.

## Review

Drafts are never auto-merged. The dashboard's draft view (see [Drafts](/translations/drafts)) shows source, proposed value, description, and an optional **back-translation** for sanity-checking locales you don't speak.

A typical review session:

1. Skim the back-translations for the high-traffic keys (top 50 by `last_seen_in_code`).
2. Read every translation in a locale you _do_ speak; trust them as a sample.
3. Spot-check 10–20 random keys in locales you don't.
4. Push tricky ones to a human translator (mark for translator → external workflow).
5. Bulk-approve the rest.

For high-stakes copy (legal disclaimers, paid plans, anything regulated) — don't trust the AI unsupervised. Use the AI draft as a starting point and have a human lawyer/translator finalise. The dashboard supports a `requires_human` flag on individual keys; rows so flagged cannot be approved by the AI's bulk actions.

## Limitations

<Callout type="warn" title="Plurals and gender are imperfect">
  ICU MessageFormat plurals are preserved syntactically but the AI may pick wrong plural forms in
  low-resource languages. Always review plural keys. ShipEasy Translations validates the category set is
  complete, but cannot validate the actual word choice.
</Callout>

<Callout type="warn" title="Length budgets are nudged, not enforced">
  A button that's 6 characters in English may be 14 in German, breaking your layout. Add length
  hints in the description (<code>max 12 chars</code>) and ShipEasy Translations's prompt nudges Claude toward
  fitting them — but verify in screenshots. The runtime can render a length-warning overlay in
  development with <code>{`<I18nDevtools />`}</code>.
</Callout>

<Callout type="warn" title="Brand-specific terms drift without a glossary">
  Without an entry in the glossary, _"flag"_ may be translated as _"drapeau"_ (literal) when you
  wanted _"feature gate"_ unchanged. Glossary first.
</Callout>

<Callout type="warn" title="Rate limits live in the upstream API">
  Anthropic enforces tier-based rate limits. Hosted plans pool a high-tier key; BYOK runs at your
  own tier. The CLI handles 429s with exponential backoff, but a 10,000-key run on a tier-1 key will
  queue. Upgrade your Anthropic tier or chunk the run.
</Callout>

## Bring-your-own-translator workflows

If you have an in-house or contracted human translation team, you can still use ShipEasy Translations as the system of record. Two patterns:
