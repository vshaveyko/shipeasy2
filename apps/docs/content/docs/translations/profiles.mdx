A **profile** is a named, versioned manifest of label keys for a specific locale combination. It is the unit you publish, version, diff, and roll back. Everything else in ShipEasy Translations — drafts, AI translations, the loader script, validation in CI — pivots around the profile.

If gates and configs are "one row per setting", profiles are "one snapshot per `locale` per publish". Every publish is immutable; revert is just "republish version N".

## Naming

Profiles use `locale`:

| Profile      | Meaning                          |
| ------------ | -------------------------------- |
| `en`    | English, production              |
| `fr`    | French, production               |
| `de:staging` | German, staging                  |
| `ja:dev`     | Japanese, dev                    |
| `pt-BR:prod` | Brazilian Portuguese, production |

The locale is BCP-47 (`en`, `fr`, `pt-BR`, `zh-Hant`). 

<Callout type="info" title="Why region tags matter">
  Use a region tag (`pt-BR`, `es-MX`, `zh-Hant`) whenever register, vocabulary, or script differs
  from the base language. The AI translator switches its system prompt based on the region tag —
  `es-MX` produces different copy than `es-ES`, and `zh-Hant` writes traditional characters where
  `zh-Hans` writes simplified.
</Callout>

## Anatomy of a profile

Every profile is a small JSON manifest in your project plus a published bundle in KV:

```json
{
  "id": "prof_01HZ...",
  "name": "fr",
  "locale": "fr",
  
  "role": "target",
  "source_profile_id": "prof_01HZ_en_prod",
  "current_version": 17,
  "key_count": 1284,
  "missing_count": 12,
  "stale_count": 4,
  "published_at": "2026-05-02T11:42:18Z",
  "loader_url": "https://cdn.shipeasy.ai/translations/v1/proj_abc/fr_prod/v17/index.json"
}
```

The `loader_url` is what the runtime hits. It is pinned to the current version, served from KV through the CDN with infinite TTL, and explicitly purged on every publish.

## Lifecycle

<Steps>
  <Step title="Create">
    ```bash
    shipeasy i18n profiles create \
      --name fr \
      --locale fr \
     
    ```

    On creation the profile is empty (`key_count: 0`, no version). It is registered in your project but nothing is written to KV until the first publish.

  </Step>

  <Step title="Populate">
    Either populate it via codemod (for the source locale) or via AI / manual translations (for target locales). See [Drafts](/translations/drafts) for the full review flow.

    ```bash
    # Source profile — pushed from a code scan
    shipeasy i18n push --profile en --source codemod

    # Target profile — drafted by the AI
    shipeasy i18n translate --profile fr --ai
    ```

  </Step>

  <Step title="Publish">
    Each publish creates a new immutable **version**. The current version is what the loader serves from KV. Old versions stay in the dashboard for rollback.

    ```bash
    shipeasy i18n publish --profile fr
    ```

    Publishing is two writes and one purge: write the chunked bundle to KV, write a new row to `i18n_profile_versions`, purge the CDN URL for the locale's `index.json`. End-to-end latency is typically under three seconds globally.

  </Step>

  <Step title="Diff">
    Before publishing — or after — compare any two versions:

    ```bash
    shipeasy i18n profiles diff --profile fr --from v16 --to v17
    ```

    Output lists added, removed, and changed keys with old and new values. Same view is available in the dashboard's **Versions** tab.

  </Step>

  <Step title="Roll back">
    From **Profiles → fr → Versions**, click any prior version → **Restore**. The selected version is republished as the new current version (so v17 → restore v15 → produces v18 with v15's contents). This keeps history append-only and roll-forward-only.

    ```bash
    shipeasy i18n profiles rollback --profile fr --to v15
    ```

    Rollback typically goes live in under five seconds — same path as a normal publish.

  </Step>
</Steps>

## Source vs target

ShipEasy Translations does not hard-code English as "the" source. The first profile you create on a project is implicitly the **source**: it is where the codemod pushes discovered keys, and it is the basis the AI translates _from_.

Other profiles are **targets**: they hold translations of the source's keys. You can switch the source profile in **Project → ShipEasy Translations → Settings**, but in practice every team picks one and sticks with it.

## Profile inheritance and fallback

When a key is missing from a target profile the loader falls back to the source profile :

```
loader requests fr label "checkout.cta"
  ↓ fr has no value
  ↓ falls back to source profile (en)
  ↓ returns "Pay"
```

This means an in-progress French translation never breaks the page — visitors see English for the missing strings until you publish translations. Fallbacks are also visible in the dashboard so you can target what to translate next; the **Coverage** column shows `1272 / 1284 (99.1%)` per target profile.

The fallback only crosses **locale**. `fr` does not fall back to `fr` — staging is meant to be the place where bad translations are caught.

## Per-locale profiles

Why not just one profile per locale? Because:

- Translators can work in `fr` while `fr` keeps serving last week's strings.
- New keys land in `en:dev` first; you do not need to translate them until they are promoted to staging or prod.
- An emergency copy fix in `en` does not require redoing translation review for development branches.
- Preview deployments (Vercel, Cloudflare Pages branch previews) can pin themselves to `:dev` or `:staging` so QA sees the unreviewed copy without it leaking to users.

The CLI's `promote` command copies a profile's values forward:

```bash
shipeasy i18n promote --from fr --to fr
```

This is the recommended flow: write and translate in `staging`, validate, promote to `prod`. Promotion creates a draft on the destination so you still get one final review before it ships.

## Pinning a build to a version

By default the loader fetches `loader_url`, which always points at the current version. For deterministic builds (a snapshot deploy, a marketing landing page that should not change copy after launch), pin to a specific version:

```html
<script
  src="https://cdn.shipeasy.ai/translations/loader.js"
  data-profile="fr"
  data-version="17"
  data-key="pk_live_..."
></script>
```

Subsequent publishes do not affect that page until you bump `data-version`. Useful for compliance pages, signed contracts, and translation review sign-offs.

## API

The same operations are available over HTTPS for programmatic use; the CLI is a thin wrapper over these.

<ApiTable>
  <ApiRow
    name="POST /translations/profiles"
    required
    type="endpoint"
    desc={
      <>
        Create a profile. Body: <code>{`{ name, locale, role? }`}</code>.
      </>
    }
  />
  <ApiRow
    name="GET /translations/profiles"
    required
    type="endpoint"
    desc="List profiles for the project. Returns id, name, current version, key/missing/stale counts."
  />
  <ApiRow
    name="GET /translations/profiles/:name"
    required
    type="endpoint"
    desc="Profile detail with current version and loader URL."
  />
  <ApiRow
    name="POST /translations/profiles/:name/publish"
    required
    type="endpoint"
    desc="Publish current draft set. Returns new version id and CDN purge status."
  />
  <ApiRow
    name="POST /translations/profiles/:name/rollback"
    required
    type="endpoint"
    desc={
      <>
        Republish a prior version. Body: <code>{`{ version }`}</code>.
      </>
    }
  />
  <ApiRow
    name="GET /translations/profiles/:name/versions"
    optional
    type="endpoint"
    desc="List versions kept under the current plan."
  />
  <ApiRow
    name="GET /translations/profiles/:name/diff"
    optional
    type="endpoint"
    desc={
      <>
        Diff between two versions. Query: <code>?from=v15&amp;to=v17</code>.
      </>
    }
  />
</ApiTable>

All endpoints take a personal access token or a CI key. The runtime SDK key cannot create or publish profiles — it is read-only against the loader URL.

## Limits

| Plan       | Profiles per project | Keys per profile | Versions kept |
| ---------- | -------------------- | ---------------- | ------------- |
| Free       | 5                    | 1,000            | 10            |
| Pro        | 50                   | 50,000           | 100           |
| Enterprise | Unlimited            | Unlimited        | Unlimited     |

Hitting a limit does not break published profiles — it blocks new publishes until you upgrade or prune. Old versions past the retention window are deleted from KV; the metadata row stays so you can see what was deleted.

<Callout type="warn" title="Deleting a profile is not reversible">
  Deleting a profile removes its KV bundle, all versions, and all drafts. The CDN purge is instant
  but in-flight loader requests may still receive a stale 404 for up to a minute. Always rename or
  stop traffic before deleting.
</Callout>
