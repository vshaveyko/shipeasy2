ShipEasy **Polylang** treats translations like configuration. You discover keys from source, store them on the project, generate translations (manually or via AI), review, and publish. New copy goes live worldwide in seconds via Cloudflare KV — no redeploy.

## How copy gets to the user [#how-it-works]

<Steps>
  <Step title="Codemod or scan">
    `shipeasy i18n scan ./src` (or the MCP `i18n_codemod_apply`) discovers strings and rewrites them as `t("key")`. Skipped strings (dynamic interpolations, rich HTML) land in `i18n-codemod-review.json` for manual review.
  </Step>
  <Step title="Push keys">
    Discovered keys are pushed to a source profile (e.g. `en:prod`). Each key has a value, an optional description, and a stable hash so re-scans don&apos;t shuffle ordering.
  </Step>
  <Step title="Translate">
    Create a target profile (`fr:prod`). Run `i18n translate --ai` to draft missing values. Claude reads your tone and glossary on every call. Review side-by-side in the dashboard.
  </Step>
  <Step title="Publish">
    `i18n publish --profile fr:prod` writes the merged label set to KV in **chunks** (so single-key edits don&apos;t invalidate the whole bundle) and purges the relevant CDN URLs.
  </Step>
  <Step title="Loader fetches at the edge">
    A small `<script>` in your HTML fetches the right chunks for the active locale. Zero JS dependencies. Infinite CDN TTL — sub-millisecond from anywhere.
  </Step>
</Steps>

## Quick install

<InstallTabs npm="npm install -g @shipeasy/cli" />

```bash
shipeasy login

shipeasy i18n install-loader --profile en:prod  # injects the script tag
shipeasy i18n scan ./src --rewrite               # discovers & rewrites t("…") calls
shipeasy i18n push --profile en:prod
shipeasy i18n profiles create --name fr:prod --locale fr
shipeasy i18n translate --profile fr:prod --ai   # AI draft
# review the draft in the dashboard, then:
shipeasy i18n publish --profile fr:prod
```

Detailed walkthrough at [Quickstart](/i18n/quickstart).

<Callout type="info" title="BYOK or hosted, your call">
  Polylang works with your own Anthropic API key (zero markup, zero data retention) **or** the
  hosted plan with a generous free tier. Switch any time in the project settings — translations stay
  in your KV either way.
</Callout>

## Configure

The config file is plain TypeScript — no YAML, no JSON gymnastics. Most teams ship the defaults and never touch it again.

```ts
// shipeasy.config.ts
import { defineConfig } from "@shipeasy/sdk";

export default defineConfig({
  i18n: {
    source: "en",
    targets: ["es", "fr", "de", "ja", "pt-BR"],
    files: ["app/**/*.{ts,tsx}", "content/**/*.md"],

    // Brand voice — Claude reads this on every translation.
    tone: "warm, technical, never marketing-speak",
    glossary: {
      ShipEasy: "ShipEasy", // never translate
      ship: "deploy", // disambiguate
    },

    provider: { kind: "anthropic", model: "claude-haiku-4-5" },
  },
});
```

## Use it in your app

Polylang ships a thin runtime — about 1.4 kB gzipped. The `useT` hook reads the active locale from your router and returns translations from the bundle generated at build time.

```tsx
// app/(marketing)/page.tsx
import { useT } from "@shipeasy/sdk/react";

export default function Page() {
  const t = useT();
  return (
    <section>
      <h1>{t("Localize, as easy as asking Claude.")}</h1>
      <p>{t("Ship in 12 languages without ever leaving your editor.")}</p>
    </section>
  );
}
```

There is no `en.json` you wrote by hand, no string keys, no `i18n.t("hero.title")`. The English **is** the source of truth and the key. Polylang generates a stable hash and keeps every locale&apos;s bundle in sync.

<Callout type="warn" title="Don't translate user-generated content this way">
  `useT` is for static UI strings extracted at build time. For UGC (chat messages, reviews, support
  tickets), use the runtime `/translate` endpoint with the streaming SDK.
</Callout>

## Why a separate product

You could store labels as a [dynamic JSON config](/configs/values) — and for tiny apps, you should. Polylang exists because real-world i18n hits problems configs aren&apos;t built for:

- **Volume**: tens of thousands of keys. One JSON blob is unworkable.
- **Workflow**: translators are not engineers and shouldn&apos;t need access to flag tooling.
- **Discovery**: keys come and go with code; manual sync drifts immediately.
- **Performance**: chunking + lazy loading by route is critical at the upper end.
- **Stats**: you want to know which keys are unused, which translations are stale, which locales lag behind.

Polylang handles all of that and exposes a dead-simple SDK for the runtime side.
